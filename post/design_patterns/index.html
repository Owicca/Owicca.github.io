factory:

  • a method to create objects that follow an interface, this way the user does not care what object is created and how, it simply calls the interface’s methods
const (
	Cash := 1
	DebitCard := 2
)

type Payment interface {
	func Pay(sum int) (string, error)
}

type CashPM struct {}
func (*c CashPM) Pay(sum int) string {
	return "", nil
}
type DebitCardPM struct {}
func (*c DebitCardPM) Pay(sum int) string {
	return "", nil
}

func getPayment(t int) (Payment, error) {
	switch t {
		case Cash: return new(CashPM), nil
		case DebitCard: return new(DebitCardPM), nil
		default:
			return nil, errors.New("Unknown payment method")
	}
}

func main() {
	pm, nil := getPayment(Cash)
	if err != nil {
		log.Fatal(err)
	}
	res := pm.Pay(100)
	log.Println(res)
}

misc:

  • I’d start with them. Programming is like building with atoms. Modules are like building with reusable routines you or your team make up. Frameworks are routines some other team made up. Design Patterns are like building with wall, door, hinge.
  • versions:
    • v1. Write the code in spagetti, maybe using classes here and there, to see if it works.
    • v2. Fix bugs, and see where patterns can apply.
    • v3. Continue to fix bugs, and add enhancements, reapplying patterns when useful.
    • v4. You get to the point where you can’t just provide a band-aid solution anymore, and it’s time to refactor. Taking all the knowledge you’ve learned in the past, and build a better OO solution.
    • v5. Add enhancements, and fix bugs, reapplying patterns can apply.
    • v6. At this point, you hopefully have a Stable version, that you are just creating enhancements, fixing less bugs. You may have provided an API, and it’s easy for your clients to customize the software.:
      • state pattern
      • template method
